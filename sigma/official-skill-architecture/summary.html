核心洞察
产出可验证性是架构决策的元变量。它决定了用前置约束还是校验循环、用大写红线还是解释原因、Body 按工作流还是知识类别组织。
Prompt vs. Code 的判断法则：确定性过程 → scripts（黑盒调用）；需理解/组合的知识模式 → SKILL.md/references。首要考量是 Context Window 预算。
负面约束是主观产出的核心武器。"NEVER 用 Inter/Roboto"比"请用好看的字体"有效，因为它对抗的是模型的统计默认倾向。
Context Window 防御是四层递进体系：结构预防 → 路由过滤 → 指针分流 → 行为劝阻，覆盖内容从设计到执行的全生命周期。
Fail-Closed 是容错的基石。Grader 不仅验证预设 assertion，还主动提取隐性声明并反向审查 eval 本身的质量。
纠正的认知误区
"控制策略取决于模型能力" → 取决于产出可验证性（同一团队对同一模型用不同策略）
"明确规则用大写红线" → 主观产出无后验手段时用大写红线，对抗统计默认倾向
下一步建议
1. 实践：选择一个你真实业务场景中的 Skill，用本会话学到的架构法则从零设计一个 SKILL.md
