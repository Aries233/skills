总览：一个 Skill 的完整生命周期
从用户输入到 Claude 产出——Skill 在其中扮演什么角色？
1触发阶段：Description 如何当守门员

用户的输入到达 Claude → Claude 查看所有可用 Skill 的 name + description（Metadata 层，始终在上下文中）→ 判断是否需要加载某个 Skill。

用户输入
→
Claude 扫描所有 description
→
匹配 → 加载 SKILL.md body
→
按需读取 references/scripts
→
产出
Description 是分类器的决策边界。Claude 天然倾向 undertrigger，所以 description 要写得"pushy"——主动列举同义词和关联场景。
2加载阶段：Progressive Disclosure 三层体系

内容按需逐层加载，控制 Context Window 消耗：

第一层 · 始终加载 · ~100 words
Metadata：YAML frontmatter 中的 name + description。每次对话都在上下文中，是触发的唯一依据。
第二层 · 触发时加载 · <500 行
SKILL.md body：Markdown 格式的完整指令。包含工作流程、约束规则、知识模式。是 Skill 的核心价值。
第三层 · 按需加载 · 无限制
Bundled resources：references/ 下的文档、scripts/ 下的脚本、assets/ 下的模板。Scripts 可在不加载源码的情况下直接执行。
3产出阶段：控制策略如何决定 Skill 设计
前置约束型 Skill
适用：主观产出（设计、写作）
策略：在编码前收窄审美空间
手段：大写红线 + 负面清单
示例：frontend-design（43行）
校验循环型 Skill
适用：可客观验证产出（代码、数据）
策略：测试 → 打分 → 迭代
手段：Grader + Benchmark + 人工审查
示例：skill-creator（485行）
判断标准只有一个：你能不能写出一个机械的 assertion 来判断输出是否正确？
SKILL.md 解剖结构
一个 Skill 的标准文件结构与每个部分的职责
标准目录结构
skill-name/
├── SKILL.md          ← 必须。YAML frontmatter + Markdown 指令
│   ├── YAML Frontmatter    → 触发机制（name + description）
│   └── Markdown Body       → 核心指令（工作流 / 知识库）
│
├── scripts/          ← 可选。确定性执行的自动化脚本
│   └── *.py / *.sh         → Claude 当黑盒调用，不读源码
│
├── references/       ← 可选。按需加载的知识文档
│   ├── aws.md              → 多领域变体文件
│   ├── gcp.md              → Claude 只读匹配的那一个
│   └── azure.md
│
└── assets/           ← 可选。模板、图标、字体等静态资源
YAML Frontmatter 字段详解
---
name: skill-name                    ← Skill 标识符
description: >                      ← 【核心】触发决策的唯一依据
  What the skill does AND when to use it.
  Make it "pushy" — 列举所有触发场景的同义词。
  即使没有明确提到关键词，相关场景也应触发。
license: Complete terms in LICENSE.txt ← 许可声明
---
description 是整个 Skill 的"守门员"。写得太窄 → undertrigger（该触发时不触发）；写得太宽 → 误触发。官方用 train/test split + 自动迭代来优化这个字段。
Body 组织方式：两种模式
工作流模式
适用：用户任务有明确的"从 A 到 B"流程
Body 是：指令序列
示例：skill-creator、mcp-builder
## Capture Intent
## Interview & Research
## Write the SKILL.md
## Run & Evaluate
## Improve the Skill
知识库模式
适用：用户每次需要不同的能力组合
Body 是：参考手册
示例：pdf、claude-api
## Python Libraries
### pypdf
### pdfplumber
## Command-Line Tools
## Common Tasks
## Quick Reference Table

规则：同一个 Skill 不应混用两种模式。如果既有强流程又有大量知识，把知识下沉到 references/。

前置约束型
投资编码前的约束声明：
• 审美方向锁定（BOLD/最小）
• 负面清单（NEVER 用 X）
• 大写红线（CRITICAL/IMPORTANT）
• 正面品质要求

例：frontend-design
写作风格选择：解释 why vs. 大写红线

同样由产出可验证性决定。主观产出需要更强的前置信号来对抗模型的统计默认倾向。

软性风格 · "解释 why"
何时用：产出可客观验证
逻辑：错了能被检测到，所以前置约束可以温和
优势：Claude 能内化原理并泛化
Keep SKILL.md under 500 lines ← 平铺直叙即可
硬性红线 · CRITICAL / NEVER
何时用：产出主观、无法后验
逻辑：模型有统计默认倾向（如 AI slop），温和指令无法打破
本质：用认知冲击对抗强先验
NEVER use Inter, Roboto, Arial ← 必须大写
负面约束模式（Negative Constraint Pattern）

当"好产出"无法定义时，退而定义"什么是坏的产出"：

反面示例 — 正面规定（不可行）
"请使用好看的字体"
"请设计有特色的页面"
"请保持高审美水准"
→ 过于抽象，模型仍会回到默认模式
正面示例 — 负面清单（有效）
"NEVER 用 Inter、Roboto、Arial"
"NEVER 用紫色渐变 + 白色背景"
"NEVER 多次生成收敛到相同选择"
→ 精确列举已知的失效模式
Prompt vs. Code 边界
什么写在 SKILL.md 里，什么封装成脚本？三层判断法则。
三层判断法则
法则 1
Context Window 预算优先
SKILL.md 有 500 行硬约束。每行每次触发都消耗 token。超出 → 分层/分流。
→
法则 2
确定性过程 → scripts/
不需要 LLM 理解内部逻辑。Claude 只需知道"何时调、传什么参数"。黑盒调用。
→
法则 3
知识模式 → SKILL.md / refs
需要 LLM 理解后灵活组合。Claude 需挑选、修改、拼接多个片段来应对具体需求。
源码对照：两种代码在 Skill 中的不同角色
scripts/ 里的代码 — 确定性执行
文件：webapp-testing/scripts/with_server.py
Claude 的角色：黑盒调用者
调用方式：python scripts/with_server.py --server "npm run dev"
官方指令："DO NOT read the source"
→ 不需要 Claude 修改哪怕一行
SKILL.md 里的代码 — 知识模式
文件：pdf/SKILL.md 内嵌的 pypdf 代码片段
Claude 的角色：理解、挑选、组合、修改
使用方式：从多个片段中选取需要的，组合改参数
如：pdfplumber 提取表格 + pandas 转 DataFrame + to_excel 导出
→ 每次用户需求不同，组合方式不同

生产级防御体系
Context Window 防御 + 容错机制 + 安全审计
Context Window 溢出：四层递进防御

覆盖内容从 设计 → 加载 → 按需读取 → 执行 的全生命周期：

第 1 层 · 结构预防 — 设计阶段
500 行硬约束 + 目录分离。
防御：Skill 作者把内容写太多。
示例：SKILL.md 保持 <500 行；reference 文件 >300 行时加目录
第 2 层 · 路由过滤 — 加载阶段
语言/项目文件探测，只加载匹配的变体。
防御：Claude 把 8 种语言的文档全加载到上下文。
示例：claude-api 检测 *.py → 只读 python/；检测 package.json → 只读 typescript/
第 3 层 · 指针分流 — 按需阶段
"See REFERENCE.md" 替代内嵌内容。
防御：低频内容每次触发都占用 Context。
示例：pdf/SKILL.md 只放 Quick Start，高级功能指向 reference.md 和 forms.md
第 4 层 · 行为劝阻 — 执行阶段
"DO NOT read the source"。
防御：Claude 执行时出于好奇去读大文件源码。
示例：webapp-testing 的 with_server.py 被当作黑盒，Claude 只用 --help 查看用法
容错机制：Fail-Closed + 深度验证
Fail-Closed 原则
存疑时，举证责任在通过方。

"误杀"（正确输出标为失败）→ 最多多改一轮
"放过"（错误输出标为通过）→ 带虚假信心发布

宁可误杀，绝不放过。
Grader 深度验证
超出预设 assertion 的深度防御：

① 主动提取隐性声明并独立验证
② 反向审查 assertion 本身是否太弱
③ 检查通过是否仅因巧合而非真正完成

三 subagent 协作：Grader + Analyzer + Comparator
安全审计：策略 + 实施双层防御
策略层
Principle of Lack of Surprise
信任边界守卫。Skill 内容不应让用户意外。禁止恶意 Skill 设计。
+
实施层
Provider Marker 扫描
作用域边界守卫。检测 import openai 等标记，防止在非 Anthropic 项目上动手。
=
效果
覆盖已知 + 未知威胁
原则覆盖新型威胁，检查覆盖已知模式。单独任何一个都不够。
跨环境适配：单文件多环境策略

三种运行环境共享 90% 核心逻辑，差异点叠加在独立段落：

## 核心工作流（所有环境共享）     ← 90% 内容，写一次
├── Capture Intent
├── Write the SKILL.md
├── Run & Evaluate
└── Improve the Skill

## Claude.ai-specific instructions   ← 无 subagent，串行测试
## Cowork-Specific Instructions      ← 无浏览器，用 --static 输出
(默认段落 = Claude Code)              ← 完整能力
单文件优于多文件：① 核心逻辑不重复写 ② 触发机制只有一个入口 ③ 环境段落仅 10-15 行，不浪费 Context。

速查表：关键架构决策
你要做的决策判断标准答案
Body 按什么组织？用户任务是否遵循固定流程？流程 → 工作流模式；按需 → 知识库模式
用前置约束还是校验循环？能否写机械 assertion 判断输出？能 → 校验循环；不能 → 前置约束
一段代码该放 SKILL.md 还是 scripts/？Claude 是否需要理解并灵活组合？需要 → SKILL.md；不需要 → scripts/
要不要用 MUST/NEVER 大写？产出能否后验验证？不能 → 大写红线；能 → 解释 why
Reference 用独立目录还是平铺？变体数 × 每变体内容量？大 → 独立目录；小 → reference/ 平铺
多环境支持拆文件还是合并？核心逻辑是否相同？相同 → 单文件多环境；不同 → 独立 Skill
SKILL.md 超过 500 行怎么办？哪部分不是每次都需要？下沉到 references/，Body 只留骨架和指针
Description 怎么写？是否列举了所有触发场景？写得"pushy"些，列举同义词，即使用户没明确提到也应触发
核心原则清单
1. Progressive Disclosure — 内容按需逐层加载，绝不一次性全部灌入上下文
2. Output Verifiability First — 产出可验证性是架构决策的元变量，决定一切下游选择
3. Context Window Budget — 每行都有成本。500 行硬约束是第一优先级
4. Negative Constraints — 无法定义"好产出"时，定义"坏产出"。对抗统计默认倾向
5. Explain Why > Shouting MUST — 解释原因比堆砌 MUST 更有效，除非在对抗默认倾向
6. Black-box Scripts — 确定性执行封装成脚本，Claude 当调用者而非读者
7. Fail-Closed — 存疑时默认失败。误杀的代价远低于放过
8. Defense in Depth — 不依赖单一防御手段。策略层 + 实施层同时存在
