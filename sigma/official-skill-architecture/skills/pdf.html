? 为什么 PDF 需要拆成 3 个 Markdown + 8 个脚本，而 frontend-design 只需要 1 个 43 行的 SKILL.md？
答案在于两者的 任务谱系 完全不同——这是你之前学过的 Output Verifiability 原则的直接应用。
Skill任务特征产出可验证性架构选择理由

生成前端页面		 不可验证 单文件纯约束 产出不可验证 → 全靠前置约束；不需要工具链

设计理由
单一主观产出用前置约束控制质量；多种客观操作用分层加载 + 脚本校验保证正确性。两者的架构模式差异源于产出是否可以被客观校验。

设计理由
PDF 操作场景极多（合并、拆分、提取、创建、OCR、加密、填表…），如果全部塞进一个文件，每次触发都加载全部内容，大部分 token 浪费在当前不需要的场景上。三文件分层 = Context Window 预算管理。
1 SKILL.md — 主入口速查手册（314 行）
覆盖 80% 常见任务的"快速路径"。包含 pypdf、pdfplumber、reportlab 的核心操作代码片段 + Quick Reference 速查表。
关键设计：
每段代码自包含、可直接复制执行 Quick Reference 表格：任务 → 最佳工具 → 一行命令 Next Steps 路由指针指向 reference.md / forms.md 特殊约束：ReportLab 上下标禁止用 Unicode
设计理由
核心路径覆盖 80% 用例，边缘路径按需加载。如果把 OCR、表单、JavaScript 库全部塞进 SKILL.md，每次合并两个 PDF 也要加载 1500+ 行无关内容——token 预算浪费，关键指令被稀释。
2 reference.md — 高级参考（612 行）
覆盖剩余 20% 边缘/高级场景。按需加载，不会污染日常操作的 Context Window。
pypdfium2 — Chromium PDFium 绑定，高速渲染 pdf-lib / pdfjs-dist — JavaScript 库操作 poppler / qpdf 进阶 CLI 用法 批量处理 · 性能优化 · 故障排查
设计理由
触发机制：SKILL.md 末尾的 Next Steps 是"路由指针"——"For advanced pypdfium2 usage, see REFERENCE.md"。LLM 在快速路径受阻时，顺着指针找到深度资源。不是把字典和百科全书印在同一本书里。
3 forms.md — 表单专项流程（295 行 · CRITICAL）
完整的表单填充 Decision Tree。强制顺序执行，三条分支路径。
CRITICAL: You MUST complete these steps in order. Do not skip ahead to writing code.
【译文】关键：你必须按顺序完成这些步骤。不要跳过直接写代码。
这行 CRITICAL 声明的作用：对 LLM 跳步倾向的显式防御。表单填充是多步骤分支流程，跳步 = 走错分支 = 产出损坏。
设计理由
为什么独立成文件而不是 SKILL.md 里的一个 section？

Decision Tree 的完整性保护——独立文件的隐含语义是"这是一个原子操作，要么完整执行，要么不执行"。如果放在 SKILL.md 里作为 section，LLM 容易在阅读其他 section 时"分心"，跳过关键的 check_fillable_fields 路由步骤。

诊断组 — 路由与校验

?
check_fillable_fields.py 268B
依赖：pypdf · 输入：PDF · 输出：stdout 文本
判断 PDF 是否有可填充字段。这是整个 forms.md 的路由入口——决定走 fillable 还是 non-fillable 分支。整个脚本只有 11 行有效代码。
为什么是脚本
需要调用 pypdf 解析 PDF 内部结构。LLM 无法"看到"PDF 的表单字段元数据，必须通过代码读取。
✓
check_bounding_boxes.py 2.7KB
依赖：无 · 输入：fields.json · 输出：stdout 校验结果
检测 bbox 重叠和高度不足。遍历所有 label/entry bbox 对，计算矩形交集，报告错误。Fail-Closed 设计：检测到任何重叠就报告 FAILURE。
为什么是脚本
矩形相交检测是确定性算法。LLM 的空间推理不可靠——让 LLM 判断"这两个框是否重叠"等于让一个没有眼睛的人判断两张照片是否一样。LLM 不擅长的事情 → 用确定性代码兜底。

提取组 — 从 PDF 中抽信息

E
extract_form_field_info.py 4.3KB
依赖：pypdf · 输入：PDF · 输出：field_info.json
提取可填充字段的完整元数据：字段 ID、页码、坐标、类型（text/checkbox/radio_group/choice）、合法值。处理了 pypdf 的字段继承（/Parent 链）和 radio group 的嵌套结构。
为什么是脚本
需要遍历 PDF 的 /Annots 注解树，解析 /FT、/_States_、/AP 等内部字段。这些是 PDF 规范底层的二进制结构——LLM 没有直接访问 PDF 内部结构的能力。
E
extract_form_structure.py 3.9KB
依赖：pdfplumber · 输入：PDF · 输出：form_structure.json
提取不可填充表单的结构：文本标签（words + 坐标）、横线（宽度 > 页面 50% 的水平线）、checkbox（5-15px 的正方形矩形）、行边界（相邻横线之间的区域）。
为什么是脚本
pdfplumber 的字符级坐标提取是确定性操作。脚本还做了业务判断（什么样的矩形是 checkbox？宽 > 页面 50% 的线才是行分隔线）——这些启发式规则比告诉 LLM "找到所有的 checkbox"可靠得多。
I
convert_pdf_to_images.py 1KB
依赖：pdf2image · 输入：PDF · 输出：PNG 图片目录
PDF → PNG，用于视觉分析。限制最大维度 1000px，平衡精度与 token 开销。
为什么是脚本
LLM 只能"看"图片，不能"看" PDF。这是 LLM 的感知边界的补偿层——脚本把 LLM 无法直接消费的格式转为 LLM 能处理的格式。

执行组 — 写入 PDF

F
fill_fillable_fields.py 3.8KB
依赖：pypdf · 输入：PDF + field_values.json · 输出：填充后的 PDF
填充可填充表单。包含字段值校验（checkbox/radio/choice 的合法值检查）+ monkeypatch pypdf 的 /Opt 字段解析 bug。校验失败直接 sys.exit(1)。
为什么是脚本 — 且含 monkeypatch
库的已知缺陷 → 封装在脚本层对 LLM 透明。pypdf 解析 /Opt 字段时返回嵌套列表而非扁平值，这是库的 bug。官方选择在脚本里打补丁而不是告诉 LLM "记得处理这个 bug"——因为 LLM 不应该在运行时做库的 bug 兼容，这属于确定性工程问题。
F
fill_pdf_form_with_annotations.py 3.2KB
依赖：pypdf · 输入：PDF + fields.json · 输出：注解叠加后的 PDF
用 FreeText 注解填充不可填充表单。自动检测 pdf_width vs image_width 关键字，选择对应的坐标转换函数。
为什么是脚本 — 双坐标系转换
PDF 坐标 y=0 在底部，图片坐标 y=0 在顶部。同一个填写操作要兼容两种来源。设计模式：LLM 只负责标注来源（用哪个关键字），脚本负责转换逻辑。不让 LLM 做数学计算。

验证组 — 可视化确认

V
create_validation_image.py 1.2KB
依赖：PIL · 输入：fields.json + 页面图片 · 输出：标注图片
在图片上画出 bbox：红色 = 填写区，蓝色 = 标签区。用于填写前的可视化验证。
为什么是脚本
填写前确认 bbox 定位是否准确——不可逆操作前的可视化验证手段。直接修改 PDF 是不可逆的，错位了用户可能不打开就发现不了。
表单填充 Decision Tree
PDF skill 最有架构价值的部分 — 一个完整的分支决策树，覆盖现实中 PDF 表单的三种形态
为什么需要这么复杂的分支树？
PDF 表单在现实中存在三种形态，每种需要完全不同的处理策略：
形态比例（经验值）处理方式
有可填充字段（fillable）~40%直接写字段值
无字段但文本结构可提取~40%结构坐标定位 + 注解叠加
扫描件/纯图片~20%视觉估算 + Zoom Refinement
设计理由
为什么不让 LLM 自己判断用哪种方式？因为 LLM 有强烈的"简化倾向"——拿到一个 PDF，它会倾向于用同一种方式处理所有情况。forms.md 的设计是强制 LLM 先用 check_fillable_fields.py 路由，然后严格按照分支执行。

为什么优先尝试结构提取
结构提取的坐标精度远高于视觉估算。PDF 内部的字符坐标是数学精确的，而视觉估算依赖 LLM 对图片的像素级判断。
无标签

设计模式：Graceful Degradation
先尝试高精度方法，失败后降级到低精度方法。Zoom Refinement 是对低精度的二次补偿——裁剪局部区域放大后再判断，减少全局坐标估算的误差。
部分
→ Hybrid：A + B 混合

统一转换为 PDF 坐标系：pdf_x = image_x × (pdf_width / image_width)
核心设计洞察
先尝试高精度方法，失败后降级到低精度方法——这是经典的 Graceful Degradation 模式。而为什么每次填写后都要 convert_pdf_to_images + 视觉验证？因为 LLM 无法直接"看到" PDF 的渲染结果。对不可逆操作（直接修改 PDF），必须提供验证手段。
Prompt vs Code 边界
从 PDF skill 提炼的三条通用法则 — 什么放 Prompt，什么封装成脚本
写在 Prompt 里（知识模式）
工具选择决策："什么任务用什么库"
代码片段模板：可直接复制执行的样板代码
约束规则：如 ReportLab Unicode 上下标陷阱
性能优化策略：大文件用 streaming 等

三条通用法则

1 LLM 做"判断"，脚本做"执行"
PDF skill 中的体现
LLM 决定"这个 PDF 是 fillable 的，用 extract_form_field_info.py"，但不参与字段值的编码转换。
LLM 决定"这个字段应该填 Smith"，但不参与将 "Smith" 写入 PDF 二进制结构的过程。
不遵循的后果
让 LLM 负责坐标系转换 → Y 轴方向搞反 → 文字写到页面外面。让 LLM 猜 checkbox 的合法值 → 写入错误编码 → PDF 损坏。
通用化
设计 Skill 时问自己："这一步 LLM 做错了后果严重吗？" 如果答案是"是"，就封装成脚本。
2 LLM 做"标注"，脚本做"转换"
PDF skill 中的体现
LLM 在 fields.json 里写 "image_width": 1700 或 "pdf_width": 612，脚本 fill_pdf_form_with_annotations.py 根据这个关键字自动选择坐标转换函数。

# fill_pdf_form_with_annotations.py:55-58 if "pdf_width" in page_info: transformed = transform_from_pdf_coords(...) # Y轴翻转 else: transformed = transform_from_image_coords(...) # 缩放+翻转
【译文】如果 page_info 中包含 "pdf_width" → 用 PDF 坐标转换（仅 Y 轴翻转）；否则 → 用图片坐标转换（缩放+翻转）。
不遵循的后果
让 LLM 自己做坐标转换 → PDF 坐标 y=0 在底部，图片 y=0 在顶部 → LLM 混淆方向 → 所有文字 Y 坐标偏移。
通用化
不要让 LLM 做数学计算或格式转换。让 LLM 提供原始信息（标注来源），脚本处理变换。
3 每一步都要有"验收门槛"
PDF skill 中的体现
填写前：check_bounding_boxes.py 校验 bbox 不重叠、高度够用
填写时：fill_fillable_fields.py 校验字段值合法，否则 sys.exit(1)
填写后：convert_pdf_to_images.py 转图片 → 人工确认
不遵循的后果
错误值静默写入 PDF → 用户不打开不发现 → 打开时文字重叠或位置错误 → 不可逆。
通用化
对于多步骤流程：在"不可逆操作"之前加校验点，在"不可见操作"之后加验证点。
6 条可复用设计原则
每条包含：原则名称 → PDF skill 中的体现 → 不遵循的后果 → 通用化到其他 Skill 的场景
1
Progressive Disclosure — 渐进加载
体现 SKILL.md（314行）→ reference.md（612行）→ forms.md（295行），按需深入
不遵循 每次触发加载 1200+ 行全部内容，token 浪费 + 关键指令被稀释
通用化 任何覆盖 3 种以上操作类型的 Skill 都应分层。每层的触发条件要明确写在上一层末尾（如 Next Steps 路由指针）
2
CRITICAL 强制顺序 — 防跳步
体现 forms.md 开头的 CRITICAL: You MUST complete these steps in order
不遵循 LLM 跳过 check_fillable_fields 直接填表 → 走错分支 → 产出损坏
通用化 任何多步骤流程型 Skill 的入口都应加强制顺序声明。CRITICAL 关键词比普通指令的遵从率高得多
3
Fail-Closed 校验 — 不信任 LLM 产出
体现 fill_fillable_fields.py 的 sys.exit(1) + check_bounding_boxes.py 的重叠检测。任何校验失败就停止，不尝试"继续试试看"
不遵循 错误值静默写入 PDF → 用户不打开不发现 → 文字重叠或字段错乱
通用化 在 LLM 产出直接修改文件的场景，必须加确定性校验。Fail-Closed（有错就停）优于 Fail-Open（有错也继续）
4
Graceful Degradation — 双路径降级
体现 Non-fillable 分支里的三级降级：结构提取（高精度）→ 视觉估算 + Zoom（低精度）→ 混合方案
不遵循 扫描件 PDF 无法处理（只用结构提取），或高精度场景被迫用低精度方法（只用视觉估算）
通用化 任何输入质量不确定的 Skill 都应设计降级路径。先尝试最优方法，失败后 fallback，而不是假设所有输入都是理想状态
5
验证闭环 — 不可见操作的补偿
体现 填写 → convert_pdf_to_images → 人工确认。scripts 里还有 create_validation_image.py 用于填写前可视化 bbox
不遵循 坐标偏移等错误无反馈通道，直到用户打开 PDF 才发现——而这时文件已经被修改了
通用化 LLM 无法直接感知操作结果的场景，必须提供中间态可视化。原则：不可逆操作前校验，不可见操作后验证
6
库缺陷封装 — 对 LLM 透明的补丁
体现 fill_fillable_fields.py 的 monkeypatch pypdf（第 74-87 行），修复 /Opt 字段解析返回嵌套列表的 bug
不遵循 要么在 Prompt 里解释 pypdf 的 bug（增加 LLM 认知负担，可能被忽略），要么让 LLM 踩坑后报错（用户体验差）
通用化 底层库的已知问题应在脚本层屏蔽，不让 LLM 感知。LLM 只需要知道"调用这个脚本"，不需要知道"这个脚本里有个补丁"
★ 一句话总结
PDF skill 的架构 = 知识分层加载（三文件 Progressive Disclosure）+ 流程强制路由（CRITICAL Decision Tree）+ 确定性脚本兜底（Fail-Closed + 验证闭环）。

这三者构成了一个不依赖 LLM "自觉性"的安全网——即使 LLM 跳步、算错坐标、不知道库的 bug，脚本层也能兜住。
自造 Skill 实战框架

0 源码级设计细节 — PDF skill 中值得逐字学习的 5 个片段

这些片段看起来平淡，但每一个都藏着官方团队的设计意图。如果自己写，大概率不会这么写。

① Description 字段的穷举式枚举
description: Use this skill whenever the user wants to do anything with PDF files. This includes reading or extracting text/tables from PDFs, combining or merging multiple PDFs into one, splitting PDFs apart, rotating pages, adding watermarks, creating new PDFs, filling PDF forms, encrypting/decrypting PDFs, extracting images, and OCR on scanned PDFs to make them searchable. If the user mentions a .pdf file or asks to produce one, use this skill.
【译文】当用户想对 PDF 文件做任何操作时使用此 Skill。包括：读取或提取文本/表格、合并多个 PDF、拆分 PDF、旋转页面、添加水印、创建新 PDF、填写表单、加解密、提取图片、OCR 扫描件。如果用户提到 .pdf 文件或要求生成 PDF，使用此 Skill。
结构 第一句 = 超宽 catch-all（"do anything with PDF files"）。 第二句 = 具体枚举 11 种操作。 第三句 = 文件扩展名兜底（".pdf"）。
设计理由
Description 是分类器决策边界。官方选择 recall > precision——宁可多触发（浪费一次 context），不可漏触发（用户问"帮我合并 PDF"却没有命中）。90+ 词的长度就是为了覆盖足够多的同义表述。

设计理由
这不是给人类看的速查表，而是给 LLM 看的决策压缩器。表格格式比段落文本更适合 LLM 做精确匹配——"用户要合并" → 扫描第一列 → 命中 "Merge PDFs" → 输出 pypdf + 代码。比让 LLM 从 300 行段落中找到合并的方法快得多。
③ IMPORTANT 标注的陷阱 — 一行警告阻止一类 bug
**IMPORTANT**: Never use Unicode subscript/superscript characters (₀₁₂₃₄₅₆₇₈₉, ⁰¹²³⁴⁵⁶⁷⁸⁹) in ReportLab PDFs. The built-in fonts do not include these glyphs, causing them to render as solid black boxes. Instead, use ReportLab's XML markup tags in Paragraph objects: chemical = Paragraph("H<sub>2</sub>O", styles['Normal'])
【译文】重要：永远不要在 ReportLab PDF 中使用 Unicode 上下标字符（₀₁₂₃₄₅₆₇₈₉, ⁰¹²³⁴⁵⁶⁷⁸⁹）。内置字体不包含这些字形，会导致它们渲染为黑色方块。应改用 ReportLab 的 XML 标记标签。
结构 IMPORTANT: [禁令] → [后果] → [替代方案 + 代码]。
设计理由
这不是一般性建议，而是特定库的特定 bug 的精确规避。它被放在 ReportLab 代码片段的正上方而非 FAQ 里——因为"在使用时一定能看到"比"在 FAQ 里能找到"重要得多。设计模式：陷阱即文档，且必须贴在使用点旁边。
④ Next Steps 路由指针 — 三文件加载的触发器
## Next Steps - For advanced pypdfium2 usage, see REFERENCE.md - For JavaScript libraries (pdf-lib), see REFERENCE.md - If you need to fill out a PDF form, follow the instructions in FORMS.md - For troubleshooting guides, see REFERENCE.md
【译文】下一步：高级 pypdfium2 用法 → 见 REFERENCE.md / JavaScript 库（pdf-lib）→ 见 REFERENCE.md / 填写 PDF 表单 → 按 FORMS.md 中的说明执行 / 故障排查 → 见 REFERENCE.md
结构 四条路由指令，每条指向具体文件 + 具体场景。"follow the instructions in FORMS.md" 的措辞比 "see FORMS.md" 更强——暗示要严格执行而非随意浏览。
设计理由
这四行是 Progressive Disclosure 的路由引擎。它们位于 SKILL.md 最末尾——LLM 读完全部基础内容后才看到路由指针，此时已经知道自己的快速路径是否够用。如果不够，顺着指针走。
⑤ 脚本错误消息 — 给 LLM 看的诊断信息
# fill_fillable_fields.py 的错误消息设计 f'ERROR: Invalid value "{field_value}" for checkbox field "{field_id}". The checked value is "{checked_value}" and the unchecked value is "{unchecked_value}"' f'ERROR: Incorrect page number for `{field_id}` (got {field["page"]}, expected {existing_field["page"]})' # check_bounding_boxes.py 的错误消息 f'FAILURE: intersection between label and entry bounding boxes for `{description}` ({ri.rect}, {rj.rect})' f'FAILURE: entry bounding box height ({entry_height}) for `{description}` is too short for the text content (font size: {font_size})'
【译文】错误：复选框字段 "{field_id}" 的值 "{field_value}" 无效。选中值应为 "{checked_value}"，未选中值为 "{unchecked_value}"。 / 错误：字段 "{field_id}" 的页码不正确（传入 {page}，期望 {page}）。 / 失败："{description}" 的标签框和输入框发生重叠。 / 失败："{description}" 的输入框高度 ({height}) 对于字体大小 ({font_size}) 太小。
结构 ERROR/FAILURE 前缀 + 具体字段名 + 期望值 vs 实际值 + 修复线索。
设计理由
这些错误消息不是给终端用户看的，而是给 LLM 看的。所以它们包含了精确的修复信息——"checked value 是 X，你传了 Y"。LLM 读完错误消息后可以直接修正 field_values.json 并重试。设计模式：脚本的 stdout 是给 LLM 的反馈通道，不是给人看的日志。
1 反面对照 — "如果我写会怎么错"

假设一个社区贡献者（不了解官方模式）要写 PDF skill，以下是大概率会犯的错误，以及官方设计如何避免它们：

社区写法
一个大文件：把所有 PDF 操作（Python + JS + CLI + 表单填充）全部塞进 SKILL.md → 2000+ 行
不写脚本：让 LLM 自己做坐标转换、字段校验 → LLM 算错 Y 轴方向 → 文字写到页面外
不强制顺序：表单填充步骤写成"建议顺序" → LLM 跳步 → 走错分支
不验证：填完 PDF 直接交付 → 坐标偏移用户不打开就发现不了
不修补库 bug：不知道 pypdf 的 /Opt bug → 某些 choice 字段静默失败
vs
官方解法
三文件分层：314 行 + 612 行 + 295 行，按需加载，token 预算不浪费
8 个脚本：坐标转换、字段校验、碰撞检测全封装 → 对 LLM 透明
CRITICAL 指令："You MUST complete these steps in order" → 遵从率远高于"建议"
双防线验证：check_bounding_boxes（填前）+ convert_to_images（填后）→ 错误可捕获
monkeypatch：在 fill_fillable_fields.py 里静默修补 → LLM 无感知
核心洞察：社区写法本质上是"信任 LLM 能力"的思路——假设 LLM 能自己判断、计算、验证。官方写法本质上是"不信任 LLM 的薄弱环节"——LLM 擅长的（语义理解、代码生成）让 LLM 做，LLM 不擅长的（空间推理、精确计算、确定性校验）封装成脚本。

3 脚本接口契约 — Prompt 与代码之间的接口设计规范

从 PDF skill 提炼的 4 条"脚本接口设计约定"。你在自己写 Skill 时，脚本与 Prompt 之间也需要遵守这些契约：

约定 1：JSON 字段名 = 坐标系信号量
LLM 在 JSON 里写 "pdf_width": 612 或 "image_width": 1700，脚本据此自动选择转换函数。
通用化：当同一操作需要兼容多种输入格式时，用字段名做信号量，而非加一个 "coordinate_system" 枚举字段。字段名本身比显式声明更不容易被 LLM 填错。
约定 2：stdout 是给 LLM 的反馈通道
脚本的 print() 不是日志，是给 LLM 的诊断信息。格式：ERROR: / SUCCESS: 前缀 + 具体字段名 + 期望 vs 实际 + 修复线索。
通用化：脚本的所有 stdout 都应该写成 LLM 能直接理解并据此修正的格式。不要写 "Error at line 42"——写 "ERROR: Invalid value 'true' for checkbox 'agree'. Use '/On' to check it."
约定 3：Fail-Closed + 具体诊断
校验失败时 sys.exit(1)，但退出前打印精确的错误信息（哪个字段、什么问题、怎么修）。
通用化：永远不要让脚本静默失败。LLM 需要明确的错误信号才能重试。exit code 1 = "停下来，修复，重跑"。
约定 4：Prompt 是编排器，脚本不关心"为什么"
forms.md 负责决定"走哪条路径"、"填什么值"。脚本只负责"执行"和"校验"——它不知道也不关心用户为什么要填 "Smith"。
通用化：脚本不应该包含任何业务逻辑判断。如果一个脚本里出现了 if/else 判断"用户想要什么"，说明逻辑放错位置了——应该放在 Prompt 里，脚本只接收结果。

6 复合型 Skill 组装案例 — 从 PDF 模式出发

三个真实业务场景，展示如何把 PDF skill 的模式与其他 Skill 的模式融合：

场景 A：智能合同审核助手
需求：用户上传 PDF 合同 → 自动提取条款 → 对照法规检查 → 标记风险点 → 生成审核报告。
组装分析：
从 PDF skill 借鉴
三文件分层（合同模板库 / 法规知识库 / 审核流程 Decision Tree）+ 脚本提取合同结构 + convert_to_images 让 LLM 看到原文排版
从 frontend-design 借鉴
审核报告的排版约束（NEVER 用默认配色、结构化呈现而非大段文字）— 报告生成是主观产出，需要前置约束
关键决策
合同提取 → 用脚本（确定性：pdfplumber 提取条款坐标）
风险判断 → 用 Prompt（主观：需要法律语义理解）
报告生成 → 用约束（主观产出 + 无 eval → NEVER 约束）
场景 B：多格式数据转换器
需求：用户上传 PDF/Excel/CSV/JSON → 自动识别格式 → 转换为目标格式 → 数据校验。
组装分析：
从 PDF skill 借鉴
路由脚本（check_fillable_fields → 改为 detect_format.py）+ Quick Reference 表格（源格式 × 目标格式 → 最佳库）+ Graceful Degradation（格式检测失败 → 回退到扩展名猜测）
独特设计
不需要三文件（操作类型多但每种都很简单）→ SKILL.md 内用分节表格覆盖所有组合。脚本只做数据校验（行列数、类型匹配）。
关键决策
不需要 CRITICAL 强制顺序（转换是原子操作而非多步流程）。不需要双防线验证（转换结果可以用脚本 100% 校验）。PDF 的模式不是全部都要用——按需取用。
场景 C：自动化测试生成器
需求：读取源码 → 分析函数签名 → 生成单元测试 → 运行测试 → 根据失败结果修正。
组装分析：
从 PDF skill 借鉴
脚本接口契约（stdout 给 LLM 看的错误消息）+ Fail-Closed 校验（测试语法不合法就 exit(1)）
从 skill-creator 借鉴（未来学习）
Eval 循环（生成 → 运行 → Grader 评分 → 迭代）+ "解释 why" 写作风格（产出可验证 → 温和型）
关键决策
产出可验证（测试是否通过 = 客观标准）→ 不需要 NEVER 约束。
但需要 CRITICAL 强制顺序（必须先读源码再生成测试）→ 借鉴 forms.md 的流程控制。
