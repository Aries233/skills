skill-creator 是一个元工具——它的用户不是终端用户，而是想创建/改进 Skill 的人。核心能力覆盖 Skill 生命周期每个阶段：

3为什么它是工程标杆？
架构特征在 skill-creator 中的体现
工作流型正文不按知识类别组织，而是按执行阶段组织：捕获意图 → 写 SKILL.md → 运行和评估 → 改进 → 循环
校验循环产出可客观验证（Skill 好不好 = 测试用例通过率），所以投资了完整的评估基础设施
跨环境适配同一文件包含 Claude Code / Claude.ai / Cowork 三个环境的适配段落，90% 逻辑共享
多子代理协作Grader 评分 → Analyzer 分析 → Comparator 盲审，三层独立角色互不干扰
自动优化引擎run_loop.py 实现了训练/测试集分割 + 自动迭代的 Description 优化系统
正文组织方式
工作流型正文——按执行阶段组织，不是按知识类别。
正文结构地图（485 行分为 6 大区块）
第 1 段 · 引言 + 沟通策略（第 1-42 行）
高层级流程概述（6 个步骤）+ 用户沟通指南。特别强调根据用户技术水平调整术语——"JSON"和"assertion"（断言）这类词要在确认用户懂之后再使用。
第 2 段 · 创建 Skill（第 45-160 行）
捕获意图（4 个关键问题）→ 访谈和研究 → 写 SKILL.md → Skill 写作指南（包含渐进式披露、Skill 解剖结构、写作模式、测试用例的完整指南）。这是知识密集段落——教 Claude 如何写 Skill。
第 3 段 · 运行和评估（第 163-288 行）
最长的连续段落。5 步序列：启动所有运行 → 起草断言 → 捕获耗时 → 评分和聚合 → 读取反馈。强调"不要中途停下"——这是一个原子操作序列。
第 4 段 · 改进 Skill（第 292-322 行）
迭代循环的核心哲学：泛化 > 过拟合。4 条改进原则 + 迭代终止条件（用户满意 / 全部空反馈 / 无法进步）。
第 5 段 · Description 优化（第 333-404 行）
独立的 Description（描述字段）优化系统。4 步流程：生成评估查询 → 用户审查 → run_loop 自动优化 → 应用结果。内含训练/测试集分割防过拟合。
第 6 段 · 环境适配 + 收尾（第 420-485 行）
Claude.ai（无子代理 → 串行测试）、Cowork（无浏览器 → 静态输出）、Claude Code（完整能力）。最后重申核心循环。
为什么是工作流型而不是知识库型？

不只是"因为有多步骤"这么简单。背后是一个关键判断：

判断依据：用户每次来到这个 Skill 时，起点和终点都一样——从"我想要一个 Skill"到"我有一个可用的 Skill"。路径是确定的，只是中间的细节不同。这和 pdf Skill 完全不同——pdf 的用户每次来的需求都不同（合并、拆分、OCR、填表……），模型需要每次挑选不同的工具组合。
skill-creator 的工作流结构
用户任务："帮我创建一个 Skill"
路径：有明确的 A→B 流程
正文结构：按执行阶段排列
Claude 的角色：流程执行者 + 引导者
→ 用户从"空白"到"可用 Skill"
对比：pdf 的知识库结构
用户任务："帮我处理这个 PDF"
路径：每次需要的工具组合不同
正文结构：按知识类别排列
Claude 的角色：参考手册查阅者
→ Claude 每次挑选不同的工具组合

工程推理 如果把 skill-creator 写成知识库型（"这是断言怎么写""这是评分怎么做"），Claude 会在用户说"帮我创建一个 Skill"时不知道先做哪个后做哪个。工作流型的价值不是"教知识"，而是强加时序——先捕获意图，再写草稿，再测试，再改进。跳步是大多数 Skill 质量问题的根源。

? 为什么"沟通策略"放在正文最前面？

485 行中，前 42 行（约占 9%）既不是技能指令，也不是流程步骤，而是教 Claude 怎么和用户说话。为什么这个优先级这么高？

"The skill creator is liable to be used by people across a wide range of familiarity with coding jargon... there's a trend now where the power of Claude is inspiring plumbers to open up their terminals, parents and grandparents to google 'how to install npm'."
【译文】skill-creator 的用户覆盖了从零基础到资深开发者的整个光谱……现在有一种趋势，Claude 的能力正在激励水管工打开终端，父母和祖父母去搜索"如何安装 npm"。
设计推理：skill-creator 是所有 Skill 中用户交互最密集的——它需要和用户反复对话（捕获意图、确认测试用例、审查结果、收集反馈）。如果 Claude 一上来就用"JSON""assertion""baseline"这些术语吓跑用户，整个流程就废了。所以 Anthropic 把"沟通策略"放在所有技术指令之前——先学会说话，再学会做事。

跨 Skill 对比 frontend-design 完全不需要"沟通策略"段落——因为它不要求用户参与流程。pdf 也不需要——用户只是说"帮我处理这个文件"。只有 skill-creator 需要深度人机协作，所以只有它把沟通策略提到了最高优先级。

写作风格分析
"解释为什么"风格的教科书级示范——产出可验证，所以用温和说服代替强硬命令。
软性说服 vs. 大写红线：官方的选择

skill-creator 的产出（Skill 质量）可客观验证（通过评估通过率）。因此官方选择了"解释为什么"的温和风格：

skill-creator 的实际写法
Try to explain to the model why things are important in lieu of heavy-handed musty MUSTs. Use theory of mind and try to make the skill general and not super-narrow to specific examples. If you find yourself writing ALWAYS or NEVER in all caps, or using super rigid structures, that's a yellow flag — if possible, reframe and explain the reasoning so that the model understands why.
【译文】试着解释为什么事情重要，而非用强硬的"必须"。用心智理论，让 Skill 保持泛化而非过窄。如果你发现自己正在用全大写 ALWAYS 或 NEVER，这是一个黄旗警告。
对比：frontend-design 的写法
CRITICAL: Typography anti-patterns. NEVER use these overused fonts: Inter, Roboto, Arial, system-ui as primary identity font. NEVER use blue-purple gradient hero sections. NEVER settle for a design that looks like a generic SaaS template.
【译文】关键铁律：排版反模式。绝对不要用 Inter/Roboto/Arial 等过度使用的字体。绝对不要用蓝紫渐变大图区域。
同一个团队的同一个作者，对不同 Skill 用了完全相反的写作风格——区别不在"规则重要程度"，而在产出可验证性。有评估兜底 → 温和；无评估兜底 → 强硬。
? 写作风格决策公式（跨 Skill 通用）

从 skill-creator 和 frontend-design 的对比中提炼的决策逻辑——适用于你未来写的每个 Skill：

有评估 + 产出可验证 → "解释为什么"
评估循环会捕获错误，所以信任模型的判断力
例：skill-creator（评估通过率验证 Skill 质量）
→ 你的 Skill 如果产出可跑自动化测试，就用这种风格
无评估 + 产出主观 → NEVER 硬约束
Prompt 是唯一防线，必须在源头堵死错误方向
例：frontend-design（"好不好看"无法自动检测）
→ 你的 Skill 如果产出靠人眼判断，就用这种风格
有评估但流程步骤关键 → 混合型
对产出质量用"解释为什么"，对流程正确性用全大写
例：skill-creator 的 Cowork 适配（唯一的全大写）
→ 流程步骤错误比产出质量差更致命时，用这种风格
无评估 + 有人工审查 → 混合型偏硬
自动化尝试 + 人工兜底 + 适度硬约束
例：skill-creator 的 Description 优化（自动化 + 人工审查 HTML）
→ 可以自动化部分验证但最终靠人把关时，用这种风格
"解释为什么"风格的 5 个实例
实例 1 — 为什么不要过拟合
"The big picture thing that's happening here is that we're trying to create skills that can be used a million times... if the skill works only for those examples, it's useless."
→ 不说"绝对不要过拟合"，而是解释为什么过拟合致命
实例 2 — 为什么用全大写是黄旗
"If you find yourself writing ALWAYS or NEVER in all caps... that's a yellow flag — reframe and explain the reasoning so that the model understands why."
→ 用元认知提示代替硬规则——让 Claude 自检
实例 3 — 为什么精简 > 冗余
"Keep the prompt lean. Remove things that aren't pulling their weight."
→ 用"不够分量"这个隐喻而非"必须简洁"
实例 4 — 为什么需要心智理论
"Today's LLMs are smart. They have good theory of mind... try to actually understand the task and why the user wrote what they wrote, and transmit this understanding into the instructions."
→ 把 Claude 当作理解能力强的协作者，不是被动执行者
实例 5 — 为什么重复说明是必要的
"Repeating one more time the core loop here for emphasis..."
→ 不说"永远记住"，而是在文末重复关键循环——利用近因效应
唯一的例外：全大写用在了哪里？

485 行中只有一处用了全大写——在 Cowork 适配段落：

GENERATE THE EVAL VIEWER *BEFORE* evaluating inputs yourself. You want to get them in front of the human ASAP!
【译文】在自行评估输入之前，*先*生成评估查看器。你要尽快让人类看到结果！
为什么这里用大写？因为这是流程正确性保障——对抗 Claude 在 Cowork 环境中"跳过评估查看器直接自己评估"的倾向。即使有评估兜底，特定流程步骤在特定环境下仍需强制。
? 深层推理：为什么"解释为什么"比 NEVER 更有效？

这不是"温和 vs. 强硬"的风格偏好。Anthropic 选择"解释为什么"有三个工程层面的原因：

原因 1：自纠错能力
当模型理解了原因，它不仅能在已知场景做对，还能在未预见的新场景中自己推导出正确行为。NEVER 只能覆盖已知错误——如果用户提出了一个官方没预见的边缘场景，NEVER 就失效了。"解释为什么"让模型具备了泛化判断力。
原因 2：评估循环兜底
skill-creator 有完整的评估循环。即使模型在某个场景下判断失误，评估会捕获错误 → 人工审查会发现问题 → 下一轮迭代会修正。这意味着"解释为什么"的风险被评估循环吸收了。在无评估的 frontend-design 中，这个安全网不存在。
原因 3：模型能力信任
源码原文："Today's LLMs are smart。" Anthropic 认为当前模型足够聪明，能够理解复杂推理。把模型当"需要理解的协作者"而非"需要命令的执行者"，产出质量更高。但这个信任有前提——产出必须有验证机制。无验证 = 无信任 = 必须 NEVER。
一句话总结："解释为什么"是高信任 + 有兜底时的最优策略；NEVER 是低信任 + 无兜底时的最优策略。不是风格选择，是风险管理的工程决策。
? 为什么在尾部重复核心循环？

源码在正文最后（第 472-481 行）把核心循环又完整列了一遍：

Repeating one more time the core loop here for emphasis: - Figure out what the skill is about - Draft or edit the skill - Run claude-with-access-to-the-skill on test prompts - With the user, evaluate the outputs - Repeat until you and the user are satisfied
【译文】再次重复核心循环以示强调：弄清楚 Skill 要做什么 → 起草或编辑 → 在测试提示词上运行 → 和用户一起评估 → 重复直到满意。
设计推理：利用 LLM 的近因效应（Recency Bias）——模型对最后读到的内容印象最深。在即将开始生成前，用核心循环覆盖模型的注意力焦点。这不是"啰嗦"，是精确的注意力控制。frontend-design 用"extraordinary... don't hold back"做激励性锚定，skill-creator 用核心循环做流程性锚定——同一个心理学原理的不同应用。

步骤 1：启动所有运行（关键设计决策）

官方明确要求：含技能和基线必须在同一个回合中全部启动。

含技能运行
→ 加载 Skill 后执行
→ 保存到 iteration-N/eval-X/with_skill/
→ 测试 Skill 的实际效果
基线运行
→ 不加载 Skill 执行（或用旧版 Skill）
→ 保存到 without_skill/ 或 old_skill/
→ 提供对比基准
为什么不分开跑？"一次性启动所有内容，这样它们大约在同一时间全部完成"——并行不仅节省时间，更重要的是让同一时刻的模型行为保持可比。
步骤 2：等待时的生产力利用
Don't just wait for the runs to finish — you can use this time productively. Draft quantitative assertions for each test case and explain them to the user.
【译文】不要干等运行完成——你可以高效利用这段时间。为每个测试用例起草定量断言并向用户解释。

这个设计体现了工程思维：不让任何时间白白浪费。运行在后台跑的同时，Claude 在前台和用户讨论断言设计——等运行结束时，断言也准备好了。

步骤 3：耗时数据的唯一捕获机会
When each subagent task completes, you receive a notification containing total_tokens and duration_ms. Save this data immediately to timing.json. This is the only opportunity to capture this data — it comes through the task notification and isn't persisted elsewhere.
【译文】当每个子代理任务完成时，你会收到包含 total_tokens（总令牌数）和 duration_ms（耗时毫秒）的通知。立即保存到 timing.json。这是捕获这些数据的唯一机会——它通过任务通知传递，不会在其他地方持久化。
如果错过了通知，耗时数据就永久丢失。这体现了 Fail-Closed（失败即关闭） 思想："逐个处理通知，而不是试图批量处理"。
X 跨 Skill 模式对照：评估循环 vs 负面约束

skill-creator 的评估循环和 frontend-design 的负面约束系统，表面完全不同，本质解决同一个问题：如何确保输出质量。

维度skill-creator：评估循环frontend-design：负面约束
质量保障方式运行后验证（事后检查）运行前约束（事前预防）
成本高（需要子代理、脚本、人工审查）低（只需 SKILL.md 里的文字）
适用条件产出可客观验证产出主观，无法自动验证
发现错误的时机运行后立即发现可能运行多次后才意识到
你的 Skill 该选哪个产出是文件、数据、代码产出是设计、文案、创意
实战决策：如果你的 Skill 产出可以通过脚本验证（如"生成的文件是否包含某列数据"），就投资评估循环；如果产出只能靠人眼判断（如"这个设计好不好看"），就把精力放在负面约束上。不要在主观产出的 Skill 上硬塞评估循环——那是浪费工程投入。

? 为什么评估循环是 5 步而不是更简单？

一个更"精简"的评估可以是：运行 → 评分 → 改进，3 步搞定。为什么官方要拆成 5 步？因为每一步解决一个不同的工程问题：

步骤它解决什么问题如果省掉会怎样
步骤 1 并行启动确保含技能和基线在同一条件下对比分开跑 = 模型版本/温度不同 = 对比无效
步骤 2 等待时写断言不浪费等待时间 + 断言和运行解耦先写断言会受预设偏见影响；后写意味着干等
步骤 3 即时捕获耗时唯一能拿到耗时数据的机会错过通知 = 永久丢失 = 无法分析性能回归
步骤 4 评分+聚合+分析+查看器从不同角度审视同一份结果只评分 = 可能漏掉系统性问题；只看查看器 = 无量化依据
步骤 5 读取人工反馈人的直觉发现自动化检测不到的问题只靠自动评分 = 过拟合断言，忽略真实体验
设计哲学：5 步不是过度工程化。每一步要么防止一种特定的信息丢失（步骤 3），要么消除一种特定的偏差（步骤 1 消除时间偏差，步骤 5 消除自动化偏差）。少一步就多一个盲区。
Description 优化系统
自动化的触发精度优化引擎——用机器学习训练思维来解决 Skill 触发问题。
为什么需要专门的 Description 优化？
问题：触发不足
Claude 天然倾向不触发 Skill。即使 description 匹配，简单任务也不会触发（因为 Claude 觉得自己能搞定）。结果：精心设计的 Skill 被冷落。
解法：主动型 Description
不是写一段中性描述，而是主动列举同义词和关联场景。然后用自动化引擎测试实际触发率，迭代优化。
四步优化流程
步骤 1 — 生成评估查询
8-10 个应触发（不同措辞、正式/随意、边缘场景）+ 8-10 个不应触发（近邻域、关键词重叠但意图不同）= 共 20 个查询。

关键要求：必须真实且具体。官方明确区分好坏示例：
坏的查询
"格式化这些数据"、"提取 PDF 文本"
好的查询
"老板发了个 xlsx 叫'Q4销售终稿v2.xlsx'，让我加一列利润率百分比…"
步骤 2 — 用户审查
生成一个交互式 HTML 页面，用户可以：编辑查询、切换是否应触发开关、增删条目。用户确认后才进入优化循环。
这一步不可跳过——"坏的评估查询导致坏的描述"。
步骤 3 — 自动优化循环（run_loop.py）
60% 训练集 40% 留出测试集 最多 5 轮 每查询重复 3 次
每轮：在训练集上评估当前 description → Claude 提出改进建议 → 新 description 在训练 + 测试集上评估 → 按测试集得分（非训练集）选最佳

为什么用测试集得分选择？防止过拟合——description 可能"背下来"训练查询但泛化能力差。
步骤 4 — 应用结果
从 JSON 输出中取出 best_description（最佳描述），更新 SKILL.md 的 YAML frontmatter（元数据区）。向用户展示修改前后对比 + 各轮得分。
触发机制的本质
Skills appear in Claude's available_skills list with their name + description, and Claude decides whether to consult a skill based on that description. The important thing: Claude only consults skills for tasks it can't easily handle on its own — simple, one-step queries may not trigger a skill even if the description matches perfectly.
【译文】Skill 出现在 Claude 的可用列表中，带有名称和描述，Claude 根据描述决定是否使用。重要的一点：Claude 只在任务自己处理不了时才使用 Skill——简单的单步查询即使描述完全匹配也不会触发。
Skill 触发的底层逻辑：Claude 的决策不是"描述是否匹配"，而是"我需要帮助吗？"。"读一下这个文件"不会触发——无论 description 写得多好。评估查询必须足够复杂才能测出真实的触发行为。
? 为什么用机器学习的训练/测试分割来优化 Description？

这是整个 skill-creator 中最"学术"的设计决策。为什么不用更直觉的方式——比如人工看触发结果然后手动调 description？

直觉方式的问题
人看 3-5 个触发结果 → 调 description → 再看 3-5 个

过拟合风险极高：你会不自觉地让 description 精确匹配你手头这几个查询。但真实用户的措辞千变万化——description "背下来"了测试查询，遇到新查询反而不触发。
训练/测试分割的解决方案
60% 查询用于"训练"（指导改进方向），40% 查询完全不看（纯测试泛化能力）。

最终按测试集得分选择——如果 description 在训练集上满分但测试集不及格，说明它过拟合了，直接淘汰。这是一个从机器学习领域借用的通用防过拟合原则。
工程洞察：为什么是 60/40 而不是 80/20？因为只有 20 个查询——80% 训练 = 16 个，测试只有 4 个，统计意义不够。60/40 = 12 训练 + 8 测试，是小样本下最务实的平衡。为什么最多 5 轮？因为 20 个查询的信息量有限——超过 5 轮迭代就是在噪声上拟合。
子代理协作体系
三个独立角色，各有明确职责边界——Grader 评分、Analyzer 分析、Comparator 盲审。
三角色关系图
Grader（评分员）
agents/grader.md
每次评估运行都调用
→ 评分 + 深度验证
Analyzer（分析师）
agents/analyzer.md
两处调用：
① 基准模式检测
② 盲审后分析"为什么赢了"
→ 表面数字下隐藏的模式
Comparator（盲审员）
agents/comparator.md
可选的高级功能
→ 盲审 A/B 产出，不知哪个是哪个
每个子代理是独立上下文。Grader 看不到 Analyzer 的分析，Comparator 不知道 A/B 哪个是新版。信息隔离确保判断客观。
? 为什么是 3 个独立子代理而不是 1 个"全能评审员"？

一个更直觉的设计是：让一个代理同时负责评分、分析、对比。为什么 Anthropic 把它拆成三个独立角色？

原因 1：信息隔离 = 消除确认偏差
如果 Grader 知道 Analyzer 的结论是"Skill B 更好"，它在评分时会不自觉地偏向 B。独立上下文意味着每个代理基于原始数据做判断，不受其他代理的结论污染。这是科学实验中"双盲"原则的直接应用。
原因 2：不同任务需要不同的思维模式
Grader 需要严格、微观、逐条——像质检员。Analyzer 需要宏观、模式识别——像数据科学家。Comparator 需要主观、整体——像用户。这三种思维模式如果塞进同一个 prompt，模型会在"严格"和"灵活"之间反复切换，降低每项的执行质量。
原因 3：可选性和渐进式复杂度
Grader 是核心（每次评估都用），Analyzer 是增强（检测模式），Comparator 是可选（盲审）。拆成三个意味着用户可以渐进式采用：先用 Grader，需要更深入分析时加 Analyzer，需要严格 A/B 对比时加 Comparator。如果合成一个"全能评审员"，就是全有或全无。
你的 Skill 借鉴点：如果你需要多角度评估，不要写一个巨型 prompt——拆成独立角色，每个有自己的专用 prompt。这在 prompt 工程中叫做"角色隔离模式"（Role Isolation Pattern）。
1Grader：超越简单评分的三层验证
第一层 · 预设断言验证
对每条预期：搜证 → 判定通过/失败 → 引用具体证据。要求"反映真实实质而非表面合规"——文件存在且内容正确，不只文件名对。
第二层 · 隐性声明提取与验证
从产出中主动提取未预设的声明（"表单有 12 个字段"、"用了 pypdf 填表"）并独立验证。能捕捉预设断言漏掉的问题。
第三层 · 评估本身的质量审查（反噬）
反向审查断言是否太弱（"通过但错误输出也会通过"）或太窄（"重要结果没有断言覆盖"）。反噬评估自身——Fail-Closed 原则的极致。
Fail-Closed（失败即关闭）原则：
When uncertain: The burden of proof to pass is on the expectation. PASS when: Clear evidence + reflects genuine substance FAIL when: No evidence / contradicted / cannot verify / superficial / coincidental
【译文】不确定时：通过的举证责任在预期方。通过条件：清晰证据 + 反映真实实质。失败条件：无证据/被反驳/无法验证/表面的/巧合的。
2Analyzer：两种工作模式
模式 A：基准分析
检测 5 种断言模式：
① 总是通过 → 不区分能力，可能没用
② 总是失败 → 可能坏了
③ 有 Skill 通过 → 无 Skill 失败 → Skill 有价值
④ 有 Skill 失败 → 无 Skill 通过 → Skill 可能有负作用
⑤ 高方差 → 可能不稳定
模式 B：事后分析
盲审出胜者后，"开箱"分析为什么：
① 比较两个 Skill 的结构差异
② 比较两次执行的记录
③ 给指令遵循度打分
④ 生成优先级排序的改进建议
→ 从 A/B 结果中提炼可操作的洞察
3Comparator：盲审的严谨性
give two outputs to an independent agent without telling it which is which, and let it judge quality. Then analyze why the winner won.
【译文】把两个产出给一个独立代理，不告诉它哪个是哪个，让它评判质量。然后分析为什么赢家赢了。
盲审的价值：消除了"新版本一定更好"的确认偏差。有时候新版本实际上更差——只有盲审能发现这一点。

官方注释："可选的，需要子代理，大多数用户不需要。人工审查循环通常就够了。"

1 实战：如何为你的 Skill 搭建评估系统

从 skill-creator 的评估循环中提炼的最小可用评估方案——不需要 9 个 Python 脚本，只需要核心思路：

步骤 1 写 3 个真实测试提示词
不是"测试这个 Skill"，而是真实用户会说的话。参考 skill-creator 的好/坏查询对比。
关键：查询越具体越好——包含文件名、上下文、用户背景。
步骤 2 为每个测试写 2-3 条可验证的断言
断言必须可客观判定通过/失败，不含主观判断。
好的断言："输出的 CSV 包含'profit_margin'列且值为百分比"
坏的断言："输出的 CSV 看起来很专业"
步骤 3 含/不含 Skill 各跑一次，对比结果
用 skill-creator 的含技能/基线并行模式——只有对比才能证明 Skill 真的有价值。
即使不用脚本，手动跑两次并记录差异也行。
步骤 4 让人看结果 → 收集反馈 → 改进 → 重测
skill-creator 的核心洞察：自动化评估 + 人工审查缺一不可。
自动化能发现"对不对"，人工能发现"好不好"。
2 渐进式披露：你的 Skill 超过 500 行怎么办

skill-creator 在 SKILL.md 第 86-109 行定义了三层加载系统。这对你的复合型 Skill 至关重要：

第 1 层 · 元数据（始终加载，约 100 词）
name + description。这是触发决策的唯一依据。无论 Skill 多复杂，这一层必须精炼到 100 词以内。
你的操作：把触发相关的信息全部压进 description，不要放到正文里。
第 2 层 · SKILL.md 正文（触发后加载，理想 < 500 行）
核心指令——工作流步骤、约束规则、写作风格要求。如果超过 500 行，说明你需要第 3 层。
你的操作：正文中只放"做什么"和"怎么做"的指令，把"详细参考资料"移到 REFERENCE.md。
第 3 层 · 资源文件（按需加载，无限量）
REFERENCE.md（知识库）、scripts/（自动化脚本）、assets/（模板/图标）。模型只在需要时读取，脚本可以执行而不需要加载内容。
你的操作：在正文中明确标注"当你需要处理 AWS 部署时，读 references/aws.md"——给模型清晰的导航。
实战原则：第 1 层决定"是否触发"，第 2 层决定"怎么执行"，第 3 层提供"需要的知识/工具"。如果你的 Skill 在第 2 层就超过 500 行，检查是否有内容可以下沉到第 3 层。

5 模式组合的陷阱与对策
陷阱 1：为所有 Skill 强加评估循环
学了 skill-creator 的评估循环后，你可能想给每个 Skill 都加。但主观产出（设计、文案）的评估循环是伪科学——断言要么太严格（拒绝好作品），要么太宽松（放过差作品）。
→ 对策：参考"评估循环 vs 负面约束"对照表（本页第 1 节）。主观产出用负面约束，客观产出用评估循环。
陷阱 2：过度信任"解释为什么"风格
"解释为什么"只在有评估兜底时才安全。如果你的 Skill 无评估、产出又高风险，温和的风格会让模型在危险的地方"自由发挥"。
→ 对策：用写作风格决策公式（第 3 页）。无评估 + 高风险 = 必须 NEVER。
陷阱 3：渐进式披露变成"渐进式隐藏"
把太多内容推到 REFERENCE.md 后，模型可能不知道什么时候该去读。skill-creator 的做法是"在正文中明确标注导航"——不是藏起来，而是给地图。
→ 对策：正文中每个 REFERENCE 引用都要说明什么时候读、读完做什么。不是"参见 references/aws.md"，而是"当用户要求部署到 AWS 时，读 references/aws.md 获取平台特定配置"。
