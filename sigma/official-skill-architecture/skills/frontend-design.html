4 设计思考的四个维度
① Purpose（目的）
这个界面解决什么问题？谁在使用它？

→ 功能性锚定：确保设计服务于用途而非"好看"
② Tone（调性）
选一个极端：极致简约、极繁混乱、复古未来主义……

→ 极端化选择：不给 "中等" 选项，迫使模型做出鲜明决策
③ Constraints（约束）
技术要求（框架、性能、无障碍访问）

→ 工程边界：确保创意不违反技术约束
④ Differentiation（差异化）
是什么让这个设计令人难忘？用户会记住的那一个亮点是什么？

→ 记忆点设计：强制模型思考"这个设计的独特标识是什么"
! 铁律（CRITICAL Rule）
CRITICAL: Choose a clear conceptual direction and execute it with precision. Bold maximalism and refined minimalism both work — the key is intentionality, not intensity.
【译文】关键铁律：选择一个清晰的概念方向，并精确执行。大胆的极繁主义和精致的极简主义都可行——关键在于"有意为之"，而非"强度堆砌"。

架构洞察 这条铁律解决的是 LLM 最常见的审美失败 — 平庸的中间路线。它不是要求 "越夸张越好"，而是要求 "有明确的意图"。极简和极繁都可以，但不能是无聊的中间态。

intentionality, not intensity — 意图的明确性 > 强度的堆砌

N NEVER — 全文最强的约束词
NEVER use generic AI-generated aesthetics like overused font families (Inter, Roboto, Arial, system fonts), cliched color schemes (particularly purple gradients on white backgrounds), predictable layouts and component patterns, and cookie-cutter design that lacks context-specific character.

逐层拆解：

层级原文对抗的 AI 默认行为
字体Inter、Roboto、Arial、系统字体LLM 默认选择"安全"字体
配色白色背景上的紫色渐变LLM 最爱的"科技感"配色

N 第二条 NEVER — 防收敛
NEVER converge on common choices (Space Grotesk, for example) across generations.

精准打击 特别点名 Space Grotesk。为什么？因为在第一条 NEVER 禁掉了 Inter/Roboto 之后，LLM 会群体"迁移"到 Space Grotesk 这个"看起来独特但其实已经烂大街"的替代品。这是二级防御 — 不仅防御已知错误，还预判了纠正后的新错误。

? 跨 Skill 工程推理：为什么这里用 NEVER 而不是"解释 why"？

同一个 Anthropic 团队，在 skill-creator 中写了：

"If you find yourself writing ALWAYS or NEVER in all caps, that's a yellow flag — reframe and explain the reasoning so that the model understands why."
【译文】"如果你发现自己正在用全大写写 ALWAYS 或 NEVER，这是一个黄旗警告——重新组织语言，解释背后的原因，让模型理解为什么。"

但在 frontend-design 中，却连续用了 2 条 NEVER。为什么同一个作者做出了完全相反的选择？

维度skill-creator（解释 why）frontend-design（NEVER）
产出可验证性可客观验证（eval pass rate）主观判断（"好不好看"）
失败模式过拟合 → eval 会捕获AI 味 → 无 eval 能检测
纠正机制内置 eval 循环自动纠偏无 eval，全靠 prompt 约束
选择的风格温和说服（模型自己能想明白）硬性红线（模型自己想不明白）
决策公式：如果产出可客观验证且有自动纠正机制 → 用"解释 why"；如果产出主观且无 eval → 用 NEVER 硬约束。

这不是风格偏好，是工程判断：frontend-design 的审美失败无法被自动化检测，所以必须在 prompt 层就堵死。
A 负面约束的架构设计 — 三层防御
第 1 层 · Description（描述）层预注入
在 Frontmatter description 中就写入 "avoids generic AI aesthetics（避免千篇一律的 AI 美学）" — 即使模型只读到 description（如触发阶段），也已经收到反 AI 味信号。
第 2 层 · NEVER 硬约束
Body（正文）中用大写 NEVER + 具体枚举（字体名、配色方案）构建不可逾越的红线。模型对 NEVER 的遵从度远高于 "avoid"（避免）。
第 3 层 · 正向引导 + 防收敛
"创造性地诠释并做出出人意料的选择" + "每次设计都应该不同" — 不仅说"不要什么"，还说"要什么"（创造力 + 不可重复性），并预判纠正后的收敛行为。

I IMPORTANT（重要规则）— 适配规则
IMPORTANT: Match implementation complexity to the aesthetic vision. Maximalist designs need elaborate code with extensive animations and effects. Minimalist or refined designs need restraint, precision, and careful attention to spacing, typography, and subtle details. Elegance comes from executing the vision well.
【译文】重要规则：实现复杂度要匹配审美愿景。极繁设计需要精细的代码、大量的动画和效果。极简或精致的设计需要克制、精确，以及对间距、排版和微妙细节的细心关注。优雅来自于把愿景执行到位。

架构洞察 这条规则解决的是代码复杂度与设计意图的匹配问题。不是所有设计都需要复杂代码 — 极简设计需要的是精确的间距和排版，而不是堆砌动画。这防止了模型在"简约风"需求下仍然输出 500 行 CSS 动画的错误。

架构模式提炼
从 43 行代码中提炼出的 6 个可复用设计模式
1
极简 Skill 的充分性判定
何时只需 SKILL.md 就够了？ 当 Skill 的核心挑战不是"如何做"（流程），而是"做成什么样"（标准）时，纯约束型 Skill 就是最佳选择。frontend-design 不需要定义工作流 — Claude 天生会写前端代码。它需要的是审美标准。
判断公式：如果移除所有约束后模型仍能完成任务，只是质量不够好 → 用纯约束型
2
描述即触发
将触发条件完全嵌入 Frontmatter description，而非依赖 Body 中的条件判断。description 中穷举所有可能的用户请求变体（web components, pages, posters, dashboards...），最后用 catch-all 兜底（"styling/beautifying any web UI"）。

优势 零延迟触发 — 系统只需读 description 就能判断是否加载 Skill，不需要解析 Body。

跨 Skill 出现 skill-creator 的 description 同样采用穷举 + 哲学句式；pdf 的 description 更加入具体的工具能力枚举（"merge, split, OCR, fill forms"）。这是所有官方 Skill 的通用做法。
3
正负约束互补
只说"不要什么"（NEVER）→ 模型不知道"要什么"；只说"要什么" → 模型可能踩红线。frontend-design 的策略是负面约束划定边界 + 正面枚举填充工具箱：
负面（边界）
禁用 Inter/Roboto/Arial
禁用紫色渐变+白底
禁用可预测布局
+
正面（工具箱）
渐变网格
非对称 + 重叠
交错显示
4
预判式二级防御
当你禁止了行为 A，模型会迁移到行为 B。frontend-design 在禁止 Inter/Roboto 后，额外点名禁止了 Space Grotesk — 这是预判模型"纠正后的新错误"的二级防御。

适用场景 任何涉及"禁止默认行为"的 Skill 都应思考：纠正后模型会去哪？那个"去哪"是否也需要禁止？

跨 Skill 出现 skill-creator 的 Description 优化系统用了 train/test split 防过拟合——本质也是二级防御（防止 description "背下来" eval query）。同一个模式在不同 Skill 中以不同形态出现。
5
时序强制
"Before coding, understand the context" — 用 "Before" 这个时序词强制模型在编码前完成思考。虽然 LLM 不真正"先思考后编码"（它是单次生成的），但在 prompt 中设定时序顺序可以引导注意力分配：让模型先"读"到思考框架，再"读"到编码要求，从而在生成时自然遵循先思考后编码的模式。
6
心理锚定收尾
最后一行 "Remember: Claude is capable of extraordinary creative work. Don't hold back" 是心理锚定而非技术约束。它利用了 LLM 对最后输入的近因效应（Recency Bias）：在即将开始生成前，用最高能量的词汇（extraordinary, don't hold back）设置输出基调。

技巧 适用于所有需要高质量创意输出的 Skill — 在 prompt 尾部放置激励性语句。

0 前置知识：官方 Skill 的三种类型光谱

你未来写的每个 Skill 都落在这条光谱上。理解这条光谱是设计复合型 Skill 的基础：

纯约束型
只有 SKILL.md
核心 = 标准/风格控制
例：frontend-design
知识库型
SKILL.md + REFERENCE.md
核心 = 领域知识注入
例：pdf、claude-api
工作流型
SKILL.md + 脚本 + subagent
核心 = 多步骤流程控制
例：skill-creator

复合型 Skill 你的终极目标 — 根据业务需求从三种类型中挑选并组装模式，而非死守一种。

2 写作风格决策框架

这是 frontend-design 和 skill-creator 对比中最重要的工程洞察——同一个团队用完全相反的风格，背后的决策逻辑：

问自己这两个问题：
Q1: 产出可客观验证吗？
可以 → eval 循环会捕获错误
不可以 → prompt 层必须堵死
Q2: 有自动纠正机制吗？
有 → 用温和的"解释 why"风格
没有 → 用强硬的 NEVER 红线
Q1=可以 + Q2=有 → 温和型
用"解释 why"、"that's a yellow flag"等元认知提示
例：skill-creator
→ 信任模型 + eval 兜底
Q1=不可以 + Q2=没有 → 强硬型
用 NEVER + 具体枚举 + 二级防御
例：frontend-design
→ 不信任模型 + prompt 是唯一防线
Q1=可以 + Q2=没有 → 混合型
对流程步骤用 NEVER（如 skill-creator 的 Cowork 适配），对质量标准用"解释 why"
→ 流程正确性强硬 + 质量温和
Q1=不可以 + Q2=有（罕见）
主观产出 + 人工审查 → Eval Viewer + NEVER

→ 自动化尝试 + 人工兜底
3 复合型 Skill 的组装策略

你的终极目标不是模仿某一种 Skill 类型，而是按需组装。以下是三种典型的组装场景：

场景 A：业务报告生成器
需求：用户上传数据 → Skill 自动生成精美的业务报告（PDF/HTML），包含图表、排版、专业术语。
分析：① 需要领域知识（报告格式/行业术语）→ 知识库型 ② 需要多步骤流程（数据解析 → 图表生成 → 排版 → 导出）→ 工作流型 ③ 需要审美控制（报告不能丑）→ 纯约束型
组装方案：
SKILL.md（工作流型 Body）
按阶段组织：数据摄入 → 分析 → 图表生成 → 排版 → 导出。时序强制词控制步骤顺序。
REFERENCE.md（知识库）
行业报告模板、图表配色方案、专业术语对照表。从 pdf Skill 借鉴三文件协作模式。
Body 末尾（纯约束型段落）
排版约束（NEVER 用默认字体、配色方案枚举）。从 frontend-design 借鉴正负约束互补 + 二级防御。
scripts/（自动化脚本）
数据解析脚本 + 图表生成工具 + PDF 导出。从 skill-creator 借鉴"黑盒脚本"模式。
场景 B：API 文档自动维护
需求：读取代码仓库 → 自动生成/更新 API 文档 → 保持与代码同步。
分析：① 需要领域知识（API 文档格式规范）→ 知识库型 ② 需要多步骤流程（扫描代码 → 解析 API → 生成文档 → 验证一致性）→ 工作流型 ③ 产出可客观验证（文档与代码是否一致）→ 可建 eval
关键决策：因为有 eval 兜底，所以写作风格用"解释 why"而非 NEVER。从 skill-creator 借鉴 Grader 模式验证文档一致性。
场景 C：品牌视觉守卫
需求：确保所有 AI 生成的视觉素材（图标、配色、排版）严格符合品牌指南。
分析：① 产出主观（"是否符合品牌"）→ 无 eval ② 核心挑战是约束而非流程 → 纯约束型 ③ 模型不知道品牌指南 → 需要知识注入
关键决策：无 eval + 产出主观 → 用 NEVER 硬约束 + 二级防御。品牌指南作为 REFERENCE.md 注入。本质上是 frontend-design 的约束策略 + pdf 的知识库策略的组合。
4 渐进式组装模板

不要一上来就写复合型 Skill。按以下阶段渐进构建：

Phase 1 纯约束型 MVP（仅 SKILL.md）
写 40-60 行的 SKILL.md，包含：description 触发域 + 使命声明 + 核心约束（正面 + 负面）+ 收尾锚定。
验证：模型是否能被触发？默认行为是否被纠正？
Phase 2 加知识库（+ REFERENCE.md）
当 Phase 1 的约束不够（模型缺少领域知识）时，添加 REFERENCE.md。把"教模型知识"从"约束模型行为"中分离出来。
验证：模型是否正确使用了领域知识？约束是否与知识库冲突？
Phase 3 加工作流（Body 改为按阶段组织）
当 Phase 2 的知识不够（模型需要按精确步骤执行）时，把 Body 从知识库型改为工作流型。添加时序强制词和步骤序列。
验证：模型是否遵循了正确的步骤顺序？每步的产出是否符合预期？
Phase 4 加自动化（+ scripts/ + eval）
当 Phase 3 的流程稳定后，把重复性工作自动化：脚本处理数据、eval 循环验证质量、Eval Viewer 人工审查。
验证：eval pass rate 是否达标？人工审查是否发现新问题？
核心原则：每个 Phase 都是一个可独立验证的完整 Skill。不要跳过 Phase 1 直接写 Phase 4 — 你需要在每个阶段确认约束有效，才能在下一层构建上它。
5 模式组合的陷阱与对策
陷阱 1：NEVER 滥用
从 frontend-design 学到 NEVER 很有效后，你可能想在所有 Skill 中都用 NEVER。但 skill-creator 的经验告诉我们：当产出可验证时，NEVER 反而限制了模型的灵活性和创造力。
→ 对策：先用写作风格决策框架（本页第 2 节）判断，再决定是否用 NEVER。
陷阱 2：约束与知识库冲突
REFERENCE.md 说"用 pypdf 库"，但约束说"NEVER 用 pypdf"——模型会困惑。
→ 对策：约束层和知识层必须用同一套假设。写 REFERENCE 时同步检查约束是否与之矛盾。
陷阱 3：过早加 eval
在 Phase 1（纯约束）还没验证有效时就投入 eval 基础设施——结果 eval 测的是错误的东西。
→ 对策：严格遵循渐进式模板。Phase 1-2 用人工验证，Phase 3+ 才考虑自动化 eval。
