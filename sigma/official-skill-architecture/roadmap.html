1. Progressive Disclosure（渐进式披露）
三层加载体系的设计原理
85%
导师提问

假设你正在设计一个 Skill，目标是让 Claude 自动将 Figma 设计稿截图转化为可运行的前端代码。如果 SKILL.md 的总行数你只能写 300 行，你会怎么分配？哪些内容值得占大头，哪些应该被推到 REFERENCE.md 或 scripts/ 里去？

你的回答

会分为四层：输入、分析、构建、校验。Input 层引导用户提供截图，Analysis 层通过 MCP 分析截图，Build 层根据 UI 规则构建 HTML，Verify 层通过 Playwright 截图对比，相似率超 80% 通过，否则调整重试。

最终沉淀

官方的 Progressive Disclosure 是一个三层加载体系：

1. Metadata（name + description）— 始终在上下文中（~100 words）
2. SKILL.md body — 触发时加载（<500 行为理想）
3. Bundled resources — 按需加载（无限制，scripts 可在不加载源码的情况下执行）

关键约束：SKILL.md 不超过 500 行。Reference 文件超过 300 行时应包含目录。当 Skill 支持多个领域/框架时，在 references/ 下按变体组织，Claude 只读取相关的那个。

2. Output Verifiability 与控制策略选择
什么时候用前置约束，什么时候用校验循环？
85%
导师提问

官方 frontend-design 放弃了你设计的那种多轮校验循环，转而把篇幅投资在"编码前的审美约束"上。控制 Claude 输出质量，更有效的策略是"事后检验和纠正"，还是"事前把审美空间收窄到不可能出错"？为什么？

初始回答（后被纠正）

前置约束倾向于模型非常聪明能完全按约束执行，校验循环倾向于模型可能出错需把控。如果模型能力很强第一种效率更高，如果模型能力不稳定第二种效率更高。

导师反例

同一个 Anthropic 团队，在同一个仓库里，同时使用了两种策略：
• frontend-design：43 行，全是前置约束，零校验循环
• skill-creator：485 行，包含完整的校验闭环（subagent 测试 → grader 打分 → benchmark → 迭代）

既然底层都是同一个 Claude 模型，为什么用不同策略？

你修正后的回答

产物属性决定策略。HTML 在设计时不固定反而是一种优势，需要发散式创造。Skill 是严谨的需要固定结构输出、标准化。选择标准取决于产物的属性，更偏向严谨还是创造。

最终沉淀

官方用词不是"creative vs. rigorous"，而是 "objectively verifiable vs. subjective"。

Skills with objectively verifiable outputs (file transforms, data extraction, code generation, fixed workflow steps) benefit from test cases. Skills with subjective outputs (writing style, art) often don't need them. — skill-creator SKILL.md

判断标准：你能不能写出一个机械的断言（assertion）来判断输出是否正确？
• 能 → 投资校验循环（如 skill-creator 的 grader + benchmark 体系）
• 不能 → 投资前置约束（如 frontend-design 的审美红线）

3. Negative Constraint Pattern（负面约束模式）
用"绝对不要"清单对抗模型统计默认倾向
80%
导师引导

frontend-design 的约束里有一个精妙的二级策略——它用的是负面清单而非正面规定：不说"用什么字体"，而是说"绝对不要用 Inter、Roboto、Arial"。

最终沉淀

当"好设计"无法机械验证时，官方用列举反模式作为桥接技术：

NEVER use generic AI-generated aesthetics like overused font families (Inter, Roboto, Arial, system fonts), cliched color schemes (particularly purple gradients on white backgrounds), predictable layouts and component patterns... — frontend-design SKILL.md

负面约束的本质是对抗模型的统计默认倾向。Claude 生成前端代码时天然偏向 Inter、Roboto、紫色渐变等高频训练数据模式。温和的"请避免通用字体"无法覆盖这个强先验，需要用 NEVER 这种强信号来打破。同时，"NEVER converge on common choices (Space Grotesk, for example) across generations"确保每次生成都不同，避免模型收敛到某个"安全"选择。

4. SKILL.md 写作风格
"解释 why" vs. 大写红线，何时用哪种？
80%
导师提问

skill-creator 说"少用 MUST/NEVER，要解释 why"，但 frontend-design 却用了 CRITICAL、NEVER、IMPORTANT 大写强调。什么时候该用"解释原因"的软性风格，什么时候必须上大写的硬性红线？

初始回答（后被纠正）

对于明确的、约束性、易理解的规则用大写硬性红线，对于需 LLM 理解、推理的条件使用"解释原因"。背后的标准是内容是原则还是条件。

导师反例

frontend-design 里标了 CRITICAL 的是："Choose a clear conceptual direction"——这并不是一条"明确的、约束性的规则"，它很抽象、很主观。那为什么官方要给它标 CRITICAL？

反过来，skill-creator 里有一条很明确的规则："Keep SKILL.md under 500 lines"——一个完全可以量化的硬约束，但官方没有用大写 MUST 来强调。

一个抽象的主观要求标了 CRITICAL，一个可量化的硬约束反而平铺直叙。这和"明确规则用硬红线"矛盾了。

你修正后的回答

产出是主观类的用大写，是可客观验证类的用解释原因。因为客观类的可以进行验证，主观类的主要用前置约束，所以需要大写强调。

最终沉淀

你把 Output Verifiability 和写作风格成功关联：

主观产出无法后验验证 → 必须用更强的前置信号（大写红线）来收窄概率空间
客观产出可以后验验证 → 前置约束可以温和，因为错了能被检测到

大写红线不仅仅是"强调"，它本质上是用来对抗模型的统计默认倾向。Claude 生成前端代码时天然偏向高频训练数据模式。温和的"请避免通用字体"无法覆盖这个强先验，需要用 NEVER 这种"认知冲击"来打破。500 行硬约束不需要大写强调，因为它可以客观检查。

5. SKILL.md 完整解剖结构
Body 按工作流阶段还是知识类别组织？
85%
导师提问

skill-creator 的 Body 按工作流阶段组织（Capture Intent → Write → Test → Improve），而 pdf 的 Body 按知识类别组织（Libraries → Tools → Tasks）。这两种组织方式分别适用于什么类型的 Skill？

你的回答

分别适用于把 Skill 当作工具使用还是当作工作流使用。如果需要严格按照具体流程执行任务则选择"工作流阶段"，和流程无关、按需使用部分能力，则使用"知识类别"来组织 Body。

最终沉淀

这个区分本质上是维度二中"知识模式 vs. 确定性过程"在 SKILL.md Body 组织层面的投射：

• 工作流阶段（skill-creator、mcp-builder）：Claude 需要按顺序执行步骤，Body 是指令序列。用户的任务天然是"从 A 走到 B"的流程。
• 知识类别（pdf、claude-api）：Claude 需要按需挑选能力组合，Body 是参考手册。用户的任务每次不同，Claude 要自己判断用哪个工具。

同一个 Skill 不应该同时混用两种组织方式。如果既有强流程又有大量参考知识，应该把知识下沉到 references/，Body 只保留工作流骨架。这正是 mcp-builder 的做法。

6. Description 触发与优化机制
为什么 Description 被当作机器学习问题对待？
80%
导师提问

官方花了大量工程力气（自动优化循环、train/test split、防止过拟合）来优化一个 description 字段。为什么它值得被当作一个"机器学习问题"来对待，而不是简单写一句话就完事？

你的回答

守门员的角色，因为他决定了什么时候触发技能，如果技能都没有触发，无论技能设计得多好都没有用。

最终沉淀

"守门员"准确——不触发，一切归零。但更深层的答案是 recall vs. precision 的权衡。

Currently Claude has a tendency to "undertrigger" skills — to not use them when they'd be useful. — skill-creator SKILL.md

• Description 越宽泛 → 触发率越高 → 但可能误触发
• Description 越精确 → 误触发越少 → 但又回到 undertrigger 问题

Description 文本本质上是一个分类器的决策边界。如果把 20 个测试查询里的关键词全部塞进 description，train set 上得分高，但遇到新查询就失效——这就是过拟合。所以官方用 train/test split 选取 test score 最高的 description，而非 train score 最高的。

7. 多领域变体与 Reference 文件架构
独立目录 vs. reference/ 平铺的判断标准
80%
导师提问

claude-api 把每种语言作为独立的一级子目录（8个语言目录 + shared/），mcp-builder 把两种语言的实现指南放在 reference/ 下平铺。什么情况下用哪种？

初始回答（后被纠正）

如果是文档类的使用 claude-api 方式，如果是工具类的使用 mcp-builder 方式。

导师纠正

mcp-builder 的 reference/ 里也全是文档（TypeScript 指南、Python 指南），不是脚本。真正的区别在于数据量：claude-api 有 8 种语言 × 每种多个文件 = 30+ 个文件；mcp-builder 只有 2 种语言 × 每种一个文件 = 4 个文件。

最终沉淀

真正的判断变量是每个变体的内容量 × 变体总数：

• 变体多 × 内容量大 → 独立目录（claude-api 模式）。如果 8 种语言全放 reference/ 平铺会变成 30+ 个文件，Claude 找不到该读哪个。
• 变体少 × 内容量小 → reference/ 平铺（mcp-builder 模式）。4 个文件平铺完全够用。

这本质上是 Progressive Disclosure 的延伸——目录结构越复杂，Claude 需要越明确的导航信号来找到正确的文件。

8. Prompt vs. Code 边界法则
什么放 SKILL.md，什么放 scripts/——黑盒调用模式
85%
导师提问

webapp-testing 中官方明确禁止 Claude 读脚本源码，让 Claude 只当调用者。什么逻辑该放 Prompt 里由 LLM 灵活执行，什么动作必须下放给底层代码工具执行？

初始回答（后被深化）

与当前流程、业务相关的内容放在 prompt 中，执行过程与中间产物与业务流程不相关的放在脚本中。背后的判断条件是：过程产物是否有助于 LLM 判断。

导师揭示的两个更根本考量

考量一：Context Window 经济学。webapp-testing 禁止读源码的理由是"pollute your context window"。SKILL.md 每行在每次触发时都消耗 token 预算，脚本不会。

考量二：确定性 vs. 判断力。pdf/SKILL.md 里内嵌了大量代码片段（教 Claude 怎么用 pypdf 等），webapp-testing 则禁止读脚本源码。前者是知识（Knowledge），Claude 需要理解才能灵活应对；后者是确定性执行（Deterministic Execution），不需要 Claude 理解内部实现。

你的进一步回答

代码片段不封装成脚本是因为它们需要被理解之后灵活组合的知识。能够被固化的是重复性、固定参数、不要 LLM 进行理解只需要执行的内容。如果需要 LLM 进行理解、灵活运用，则是作为知识在 Skill 中。

最终沉淀

三层判断法则：

1. Context Window 预算优先——SKILL.md 有 500 行硬约束
2. 确定性过程 → scripts——不需要 LLM 理解，只需调用的黑盒（如 with_server.py 管理服务器生命周期）
3. 知识模式 → SKILL.md/references——需要 LLM 理解后灵活组合的代码片段（如 pdf 库的使用模式）

关键区分：pdf 里的代码片段不是"确定性过程"，它们是需要被理解后按需组合的知识模式。当用户说"提取第 3-5 页表格转 CSV"时，Claude 需要从多个片段中挑选、组合、改参数。

9. Context Window 溢出防御
四层递进防御体系
85%
导师提问

官方有四种防御 Context Window 溢出的手段：① webapp-testing 的黑盒策略 ② skill-creator 的 500 行硬约束 ③ pdf 的分流策略 ④ claude-api 的语言检测分流。这四个案例是替代关系还是递进关系？为什么需要四种？

你的回答

是层层递进的。解释性防御可能随着上下文增多而被忘记，强制约束保证加载内容始终不会很多，快速指引实现渐进性披露，限制加载逻辑保证渐进式披露。

最终沉淀

四层递进防御，每层防御不同的失效模式，覆盖内容从设计 → 加载 → 按需读取 → 执行的全生命周期：

层级 防御手段 防御的失效模式
结构预防500 行硬约束、目录分离Skill 作者把内容写太多
路由过滤语言检测、项目文件探测Claude 加载了不需要的变体
指针分流"See REFERENCE.md"低频内容每次都占用 Context
行为劝阻"DO NOT read the source"Claude 执行时好奇去读源码

10. 跨平台/跨环境适配策略
单文件多环境 vs. 多文件单环境
80%
导师提问

skill-creator 把 Claude Code / Claude.ai / Cowork 三种环境的适配逻辑全部写在一个 SKILL.md 里，而不是为每种环境写独立文件。为什么"单文件多环境"比"多文件单环境"更好？这和 Context Window 经济学是否矛盾？

你的回答

单文件多环境更好，因为核心逻辑一样，只是受限的外部环境不同。Skill 本质只是 message 文本，LLM 可以根据文本进行推理理解。即使随着上下文增多可能被遗忘，也不对主流程造成很大影响。和 Context Window 不矛盾，反而更简洁，避免同一个功能写三次，避免不被触发或重复加载。

最终沉淀

三个关键理由：

• 核心逻辑复用，差异点叠加——三种环境共享 90% 工作流，只有外设能力不同。拆成三个文件意味着 90% 的内容写三遍。
• 触发机制的单点性——三个独立 Skill 共享相似 description 会导致触发混乱或重复加载。单文件确保只有一个入口。
• 与 Context Window 不矛盾——环境适配段落占篇幅很小（每个约 10-15 行），不会造成显著的额外开销。

11. 错误处理与容错机制
Fail-Closed 原则、Grader 深度验证、隐性声明
80%
导师提问

Grader Agent 里有一条规则："When uncertain: The burden of proof to pass is on the expectation."——"存疑时，举证责任在通过方。"这体现了什么容错哲学？如果反过来——"存疑时默认通过"——会造成什么后果？

你的回答

如果默认通过，会导致 Skill 可能朝着意料之外的方向发展。对于软件设计，类似于测试工程师进行问题验证时，都是默认存疑不通过，然后通过测试用例，判断限定范围内通过，符合准出标准。

你提出的不懂的概念

你表示没有理解"反噬 eval 本身"和"隐性声明"两个概念。

概念解释

"反噬 eval 本身"：Grader 不仅要给输出打分，还要反向审查测试题出得有没有问题。例如：如果 assertion 只检查"是否生成了 .pdf 文件"（文件名存在就通过），但没有检查内容是否正确，Grader 会主动报告这个 assertion 太弱。

"隐性声明"：假设 Skill 产出了一个 PDF，Claude 的 transcript 里写了"表单有 12 个可填写字段"——这是 Claude 自己说的，不是任何 assertion 要求的。但 Grader 会主动提取这类声明并独立验证——真的有 12 个吗？如果只有 8 个，就抓出了一个 assertion 没覆盖到的错误。

最终沉淀

软件工程中的 Fail-Closed 原则（默认拒绝）：

• "误杀"（把正确输出标为失败）→ 最多多改一轮
• "放过"（把错误输出标为通过）→ 带着虚假信心发布有缺陷的 Skill

Grader 构成了超出预设检查范围的深度防御层：
① 不仅验证预设 assertion，还主动提取并验证隐性声明
② 不仅评分输出，还反向审查 assertion 本身的质量（是否太弱、是否缺乏区分度）

整个容错体系由三个 subagent 构成：Grader（打分 + 质疑 eval）、Analyzer（寻找聚合统计掩盖的 pattern）、Comparator（盲对比 A/B 测试）。

12. 安全审计边界
策略层 + 实施层的双层防御
80%
导师提问

"Principle of Lack of Surprise"（通用安全原则）和 claude-api 的 provider marker 扫描（具体技术检查）分别防御的是什么类型的威胁？为什么两者需要同时存在？

初始回答（后被精确化）

通用原则是功能层面的，具体检查是权限层面的。主要是应对现实中可能存在的安全和合规风险。

最终沉淀

Lack of Surprise 防御的是恶意 Skill 设计——有人在 Skill 里藏恶意代码。它是一个信任边界的守卫。

Skills must not contain malware, exploit code, or any content that could compromise system security. A skill's contents should not surprise the user in their intent if described. — skill-creator SKILL.md

Provider marker 扫描 防御的是范围越界——用户的项目已经是 OpenAI 生态的，claude-api Skill 不应该悄悄改写成 Anthropic 的。它是一个作用域边界的守卫。

两者必须同时存在，是安全工程里经典的策略 + 实施双层防御：
• 只有通用原则 → 太模糊，Claude 可能意识不到某个具体行为违反了原则
• 只有具体检查 → 太窄，只能拦截预定义模式，无法应对新型威胁
